# はじめに

この記事は2024年に開催された電気情報機器学(BDM)の最終課題作成の備忘録です。また、2025年五月祭のEEIC企画で展示されていた同作品の詳細記事でもあります。

# やりたかったこと

**人の顔が覚えられない!** これは、人の顔を見て話せない私の問題でしかないのだが、せっかくなので私の人間的成長以外の手段で解決してみたくなった。
そこで、人の顔を覚える、認識する、という行為を外部に委託すればいいのではないか、というモチベーションで作成された。
最終的には、ESP32-S3を用いて、メガネに装着できるカメラモジュールを作成し、PC側で顔認証を行い、LINE Notifyで通知する、というものを作成した。

# 使用したもの

## XIAO ESP32-S3 Sense

装着するにあたって小型軽量のマイコンを使用する必要があったため、ESP32を選択した。XIAO ESP32-S3 SenseとESP32-CAMなどが標準でカメラ(OV2640CMOS等)を搭載するポートがついてるため、候補にあがった。スペック比較は海外の方がYouTubeに上げてくださっていたので、そちらを参照した。

https://youtu.be/gYQDGoVRObA?si=RuicvLGegs6WKI7H

結論として、どちらもESP32-S3を搭載しているが、XIAO ESP32-S3 Sence側には8MBのフラッシュが搭載されているのに対し、ESP32-CAM側は4MBのフラッシュしか搭載されていないため、XIAO ESP32-S3 Senceを選択した。ただし、XIAO ESP32-S3 Senceは小型である分発熱がかなり大きく、また付属のアンテナ以外で技適を取得しているか不明なため、その点を考慮しながら使用する必要があった。

:::message
以降簡略のため、XIAO ESP32-S3 SenseをESP32と呼ぶことにする。
:::

## Python側

- **リアルタイム処理の負荷**: `face_recognition` ライブラリは非常に強力ですが、リアルタイムで顔認識を行うにはそれなりの計算リソースを必要とします。特にM1 MacBook AirのようなARMベースのCPUでは、IntelベースのCPUに比べて一部のライブラリの最適化が十分でない場合があり、処理速度のボトルネックとなることがありました。フレームレートを維持しつつ、複数の顔を同時に認識する際のパフォーマンスチューニングに苦労しました。
- **WebSocket通信の安定性**: ESP32側とのWebSocket通信において、ネットワーク環境やESP32の発熱による切断が頻繁に発生しました。Python側で再接続ロジックを実装したり、切断時のエラーハンドリングを丁寧に行う必要がありました。また、画像データの転送量が多くなるため、WebSocketのバッファサイズやフレーム分割の最適化も検討しました。
- **Tkinterの限界**: GUIのプロトタイプとしてTkinterを使用しましたが、よりリッチなUIや複雑なレイアウトを実現しようとすると、その表現力に限界を感じました。特に、リアルタイムの映像ストリームを表示しつつ、認識結果をオーバーレイ表示するような機能の実装は、Tkinterだけでは困難でした。

# 課題点

## ESP32側

- 発熱の問題

5min程度の稼動でもESP32が熱くなりすぎてWiFIが切断されることがあった。また、httpサーバーの起動はそのままに、カメラを休止させようとしたがそれに必要なピン(PWDN端子)がGPIOに割り当てられていなかったため、カメラを休止させることができなかった。

詳しくは以下参照
https://forum.seeedstudio.com/t/xiao-esp32s3-sense-camera-sleep-current/271258/5?page=4

- アンテナの問題

Xiao ESP32 Senceは技適の都合で付属のアンテナ以外で使用することができないため、ESP32-CAMを使用した場合と比べて通信距離が短くなってしまった。これにより、PC側からの通信が途切れやすくなってしまった。

:::details 現実的でない解決方法
電波法に基づくと、電波暗室内においては技適のないアンテナを使用しても問題ない。実際、筆者が電波暗室においてアンテナに鉄でできたクリップをつけて使用したところ、通信が大幅に改善された。しかし、電波暗室内で相手の顔がわからなくなるケースがあるとは思えないため、実用的ではない。
:::

## Python側

- デザインの問題
  Tkinterダサい。
- **顔認証の精度と登録**: `face_recognition` ライブラリは非常に便利ですが、登録されていない顔を「Unknown」として扱うため、事前に多くの顔データを登録しておく必要があります。実運用を考えると、顔の登録プロセスを簡素化したり、未知の顔を一時的に記録して後から登録できるような機能が必要だと感じました。また、照明条件や顔の向きによる認識精度の低下も課題でした。
- **通知の柔軟性**: 現在はLINE Notifyのみを使用していますが、より多様な通知方法（例: Slack、メール、カスタムAPI連携）に対応することで、利用シーンが広がると考えられます。また、通知内容も単なる「顔を認識しました」だけでなく、認識した人物の名前や、その人物に関するメモなどを付加できると、より実用的になります。
- **システム全体の統合とデプロイ**: ESP32とPythonアプリケーションを連携させる際、それぞれの起動順序やエラー時の挙動を考慮した堅牢なシステム設計が求められます。現状では手動での起動が必要ですが、将来的には自動起動やバックグラウンドでの動作、さらにはクラウドサービスとの連携なども視野に入れると、より洗練されたシステムになるでしょう。

# まとめ

本プロジェクトでは、ESP32-S3を用いたメガネ装着型カメラモジュールと、PythonによるPC側での顔認証システムを構築しました。人の顔を覚えるという個人的な課題を技術で解決するというモチベーションから始まり、小型軽量なデバイス選定、WebSocketによるリアルタイム通信、そして`face_recognition`ライブラリを用いた顔認証の実装に挑戦しました。

開発過程では、ESP32の発熱問題やアンテナの制約、Python側でのリアルタイム処理の負荷、WebSocket通信の安定性、そしてTkinterのUI表現の限界など、多くの技術的課題に直面しました。これらの課題に対し、試行錯誤を重ねながら解決策を模索し、最終的には当初の目標である「メガネ装着型カメラモジュールによる顔認証とLINE通知」を実現することができました。

今回の経験を通じて、組み込みシステムとPCアプリケーションの連携、リアルタイム画像処理、そしてUI/UXの重要性について深く学ぶことができました。特に、ハードウェアの制約とソフトウェアの要求をいかにバランスさせるかという点は、今後の開発においても重要な視点となるでしょう。

今後の展望としては、発熱問題の根本的な解決（より低消費電力なデバイスの検討や放熱設計の改善）、顔認証精度の向上と登録プロセスの簡素化、通知方法の多様化、そしてシステム全体の堅牢性とデプロイの容易化が挙げられます。これらの課題を克服することで、本システムはさらに実用的なものへと進化する可能性を秘めていると感じています。

このプロジェクトが、同様の課題に取り組む方々や、ESP32とPythonを用いたシステム開発に興味を持つ方々にとって、何らかの参考になれば幸いです。
